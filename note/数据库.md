# 数据库系统原理

# 第二章 关系模型

## 2.1 关系模型的数据结构及形式化定义

### 2.1.1 关系

#### 1. 域

域是一组具有相同数据类型的值的集合

#### 2. 笛卡尔积

给定一组域 $D_1, D_2, \dots, D_n$（域可以相同），它们的笛卡尔积是所有**有序n元组** $(d_1, d_2, \dots, d_n)$ 的集合，其中每个分量 $d_i \in D_i$（$i=1,2,\dots,n$）。

数学表达式：
$$D_1 \times D_2 \times \dots \times D_n = \{(d_1, d_2, \dots, d_n) \mid d_i \in D_i, \, i=1,2,\dots,n\}$$

### 2.1.2 关系模式

**定义 2.4** 关系的描述称为关系模式（relation schema）。它可以形式化地表示为

$$R(U,D,DOM,F)$$

其中 $R$ 为关系名，$U$ 为组成该关系的属性的属性名集合，$D$ 为 $U$ 中属性所来自的域，$DOM$ 为属性向域的映像集合，$F$ 为属性间数据依赖关系的集合。

## 2.2 关系操作

### 2.2.1 基本的关系操作

关系模型中常用的关系操作包括**查询(query)**操作和**更新操作**两大部分，而更新操作又可分为**插入(insert)**、**删除(delete)**、**修改(update)**等操作。

关系的查询表达能力很强，因此查询操作是关系操作中最主要的部分。查询操作又可进一步分为**选择(select)**、**投影(project)**、**连接(join)**、**除(divide)**、**并(union)**、**差(difference)**、**交(intersection)**、**笛卡儿积**等操作。其中**选择、投影、并、差、笛卡儿积是5种基本操作**，其他操作可以用基本操作来定义和导出，就像乘法可以用加法来定义和导出一样。

## 2.4 关系代数

关系代数是一种抽象的查询语言, 他用对关系的运算来表达查询

### 2.4.1 传统的集合运算

二目运算，含**并、差、交、笛卡儿积**4种。并/差/交要求两关系**同目同域**。

1.  **并**：$R \cup S = \{ t \mid t \in R \lor t \in S \}$，属于R或S的元组

2.  **差**：$R - S = \{ t \mid t \in R \land t \notin S \}$，属于R但不属于S的元组

3.  **交**：$R \cap S = \{ t \mid t \in R \land t \in S \}$，既属于R又属于S的元组，可由差导出

4.  **笛卡儿积**：$R \times S = \{ t_r t_s \mid t_r \in R \land t_s \in S \}$，两关系元组的所有组合，结果为n+m目

### 2.4.2 专门的关系运算

#### 1. 选择(selection)

选择又称为**限制(restriction)**。它是在关系 $R$ 中选择满足给定条件的诸元组，记作：

$$\sigma_F(R) = \{ t \mid t \in R \land F(t) = '真' \}$$

其中 $F$ 表示选择条件，它是一个逻辑表达式，取逻辑值“真”或“假”。

逻辑表达式 $F$ 的基本形式为：

$$X_1 \theta Y_1$$

#### 2. 投影(projection)

关系 $R$ 上的投影是从 $R$ 中选择若干属性列组成新的关系。记作：

$$\prod_A(R) = \{ t[A] \mid t \in R \}$$

其中 $A$ 为 $R$ 中的属性列。

投影操作是从**列的角度**进行的运算。

#### 3. 连接(join)
连接也称为 $\boldsymbol{\theta}$ 连接，指从两个关系的笛卡儿积中选取其属性间满足一定条件的元组。记作：

$$R \underset{A\theta B}{\bowtie} S = \{ \overparen{t_r t_s} \mid t_r \in R \land t_s \in S \land t_r[A] \theta t_s[B] \}$$

其中，$A$ 和 $B$ 分别为关系 $R$ 和 $S$ 上列数相等且可比的属性列，$\theta$ 是比较运算符。具体来说，连接运算是从笛卡儿积 $R \times S$ 中选取关系 $R$ 在属性列 $A$ 上的值与关系 $S$ 在属性列 $B$ 上的值满足比较关系 $\theta$ 的元组。

连接运算中有两种最为重要也最为常用的连接，一种是**等值连接(equijoin)**，另一种是**自然连接(natural join)**。

$\theta$ 为“$=$”的连接运算称为等值连接。它是从关系 $R$ 与 $S$ 的广义笛卡儿积中选取 $A$、$B$ 属性值相等的那些元组，即：

$$R \underset{A=B}{\bowtie} S = \{ \overparen{t_r t_s} \mid t_r \in R \land t_s \in S \land t_r[A] = t_s[B] \}$$

自然连接是一种特殊的等值连接。它要求两个关系中进行比较的分量必须是同名的属性列，并且在结果中把重复的属性列去掉。即若 $R$ 和 $S$ 中具有相同的属性列 $B$，$U$ 为 $R$ 和 $S$ 的全体属性集合，则自然连接可记作：

$$R \bowtie S = \{ \overparen{t_r t_s}[U-B] \mid t_r \in R \land t_s \in S \land t_r[B] = t_s[B] \}$$

一般的连接操作是从行的角度进行运算，但自然连接还需要取消重复属性列，所以是同时从行和列的角度进行运算。

#### 4. 除(division)
设关系 $R$ 除以关系 $S$ 的结果为关系 $T$，则 $T$ 包含所有在 $R$ 但不在 $S$ 中的属性及其值，且 $T$ 的元组与 $S$ 的元组的所有组合都在 $R$ 中。

下面用**象集**来定义除法。

给定关系 $R(X,Y)$ 和 $S(Y,Z)$，其中 $X$、$Y$、$Z$ 为属性列。$R$ 中的 $Y$ 与 $S$ 中的 $Y$ 可以有不同的属性名，但必须出自相同的域。

$R$ 与 $S$ 的除运算得到一个新的关系 $P(X)$，$P$ 是 $R$ 中满足下列条件的元组在 $X$ 属性列上的投影：元组在 $X$ 上分量值 $x$ 的象集 $Y_x$ 包含 $S$ 在 $Y$ 上投影的集合。记作：

$$R \div S = \{ t_r[X] \mid t_r \in R \land \prod_Y(S) \subseteq Y_x \}$$

其中 $Y_x$ 为 $x$ 在 $R$ 中的象集，$x = t_r[X]$。

# 第三章 关系数据库标准语言SQL

## 3.1 SQL概述

- 综合统一:集数据定义语言（DDL），数据操纵语言（DML），数据控制语言（DCL）功能于一体。
- 高度非过程化
- 面向集合的操作方式
- 以同一种语法结构提供多种使用方式
- 语言简洁，易学易用————9个动词:(查询)select,(定义)create,drop,alter,(操纵)delete,update,insert,(控制)grant,revoke

## 3.2 数据定义

建立数据库时首先要对数据库中数据的结构及特征进行描述，该过程称为数据定义.

在SQL语言中，关于数据的描述信息以SQL模式（Schema）的形式组织。每个模式包含0到多个对象，这些对象被称为模式对象，如基表、视图、索引等。

SQL模式语句可分为三类：CREATE语句、DROP语句和ALTER语句。

### 表：SQL 的数据定义语句

| 操作对象 | 创建 (CREATE) | 删除 (DROP) | 修改 (ALTER) |
| :--- | :--- | :--- | :--- |
| **模式 (SCHEMA)** | `CREATE SCHEMA` | `DROP SCHEMA` | |
| **表 (TABLE)** | `CREATE TABLE` | `DROP TABLE` | `ALTER TABLE` |
| **视图 (VIEW)** | `CREATE VIEW` | `DROP VIEW` | |
| **索引 (INDEX)** | `CREATE INDEX` | `DROP INDEX` | `ALTER INDEX` |

### 3.2.1 模式的定义与删除

#### 1. 模式的定义

模式定义语句如下:

```sql
CREATE SCHEMA <模式名> AUTHORIZATION <用户名>
    [<表定义子句>|<视图定义子句>|<授权定义子句>];
```

语义: 为某用户创建一个模式。定义模式实际上定义了一个命名空间，其中可以定义该模式包含的数据库对象，例如基本表、视图、索引等。

#### 2. 模式的删除

模式删除语句如下:

```sql
DROP SCHEMA <模式名> <RESTRICT|CASCADE>;
```

语义: 删除一个模式。删除模式时，如果该模式中存在对象，则必须指定删除方式：<RESTRICT> 或 <CASCADE>
- <RESTRICT>(限制): 如果模式中存在对象，则拒绝删除模式。
- <CASCADE>(级联): 删除模式及其包含的所有对象。

### 3.2.2 基本表的定义、删除与修改

#### 1. 基本表的定义

基本表定义语句如下:

```sql
CREATE TABLE <表名> (
    <列名><数据类型>[列级完整性约束条件]
    [,<列名><数据类型>[列级完整性约束条件]]
    [,PRIMARY KEY(列名[,列名]...)]
    [,FOREIGN KEY(列名[,列名]...) REFERENCES <表名>(列名[,列名]...)]
    [,UNIQUE(列名[,列名]...)]
    [,CHECK(<条件>)]
    ...
    [,<表级完整性约束条件>]
);
```

语义: 定义一个基本表。定义基本表时，必须指定表名和至少一个列定义。每个列定义包括列名、数据类型和可选的列级完整性约束条件。还可以指定表级完整性约束条件。

- <表名>: 所要定义的基本表的名字
- <列名>: 组成该表的各个属性（列）
- <列级完整性约束条件>：涉及相应属性列的完整性约束条件
- <表级完整性约束条件>：涉及一个或多个属性列的完整性约束条件
- PRIMARY KEY: 主键约束，指定一个或多个列作为主键
- FOREIGN KEY: 外键约束，指定一个或多个列作为外键，并引用另一个表的主键
- UNIQUE: 唯一约束，指定一个或多个列的值必须唯一
- CHECK: 检查约束，指定一个条件，要求列的值必须满足该条件

例子:
```sql
CREATE TABLE Course (
    Cno CHAR(4) PRIMARY KEY,          -- 课程号：主码，固定长度4字符
    Cname CHAR(40) NOT NULL,          -- 课程名：非空，固定长度40字符
    Cpno CHAR(4),                     -- 先修课号：允许为空（即没有先修课）
    Ccredit SMALLINT,                 -- 学分：使用小整数类型
    
    /* 表级完整性约束：
       定义外码 Cpno 参照本表 Course 的主码 Cno。
       这是一种“自引用”结构，表示先修课必须是课程表中已有的课程。
    */
    FOREIGN KEY (Cpno) REFERENCES Course(Cno)
);
```

### 3.2.3 索引的建立与删除

#### 1. 索引的建立

```sql
CREATE [UNIQUE] [CLUSTER] INDEX <索引名>
    ON <表名>(<列名>[<次序>][,<列名>[<次序>] ]…)；
```

其中，<次序> 可以是 ASC（升序）或 DESC（降序），默认是 ASC。

功能: 在<表名>的表上，对其中的指定列，建立一个名为<索引名>的索引文件。

作用: 提高查询速度

说明:
- UNIQUE(单一索引): 指定索引列的值必须唯一
- CLUSTER(聚集索引): 索引项的顺序与表中记录的物理顺序一致。表中如果有多个记录在索引字段上相同，这些记录构成一簇，只有一个索引值。

#### 2. 索引的修改

```sql
ALTER INDEX <索引名> RENAME TO <新索引名>;
```

#### 3. 索引的删除

```sql
DROP INDEX <索引名>；
或
DROP INDEX <索引名> ON <表名>；
```

## 3.3 数据查询

select 语句的基本格式如下:

```sql
SELECT [ALL | DISTINCT] <目标列表达式> [别名] [, <目标列表达式> [别名]] ...
FROM <表名或视图名> [别名] [, <表名或视图名> [别名]] ... | (<SELECT 语句>) [AS] <别名>
[WHERE <条件表达式>]
[GROUP BY <列名1> [HAVING <条件表达式>]]
[ORDER BY <列名2> [ASC | DESC]]
[LIMIT <行数1> [OFFSET <行数2>]];
```

SELECT语句的含义是,根据WHERE子句的条件表达式从FROM子句中指定的基本表、视图或派生表中找出满足条件的元组, 再按SELECT子句中的目标列表达式选出元组中的属性值形成结果表.

GROUP BY: 将结果按<列名1>的值进行分组,该属性列值相等的元组为一个组,通常会在每组中作用聚集函数.
ORDER BY: 按<列名2>的值对结果进行排序,默认是升序(ASC),也可以指定为降序(DESC).
LIMIT: 限制结果表中的行数,如果指定了OFFSET子句,则从结果表的第<行数2>行开始返回<行数1>行数据.

### 3.3.1 单表查询

#### 1. 选择表中的若干列

(1) 查询指定列

```sql
SELECT Sno, Sname FROM Student;
```

从Student表中取出一个元组, 取出该元组在属性Sno和Sname上的值, 形成一个新的元组.

(2) 查询所有列

```sql
SELECT * FROM Student;
```

(3) 查询经过计算的值

```sql
SELECT Sname, (extract(year from current_date) - extract(year from Sbirthday)) AS Sage
FROM Student;
```
计算当前日期与学生出生日期之间的年数差, 作为学生的年龄, 并给该列取别名Sage.

#### 2. 选择表中的若干元组

(1) 消除取值重复的行

```sql
SELECT DISTINCT Sdept 
FROM Student;
```

(2) 选择满足条件的行

通过WHERE子句指定条件表达式, 选择满足条件的行.

| 查询条件             | 谓词                                                                 |
|----------------------|----------------------------------------------------------------------|
| 比较                 | `=`, `>`, `<`, `>=`, `<=`, `!=`, `<>`, `!>`, `!<`; `NOT` + 上述比较运算符 |
| 确定范围             | `BETWEEN AND`, `NOT BETWEEN AND`                                     |
| 确定集合             | `IN`, `NOT IN`                                                       |
| 字符匹配             | `LIKE`, `NOT LIKE`                                                   |
| 空值                 | `IS NULL`, `IS NOT NULL`                                             |
| 多重条件（逻辑运算） | `AND`, `OR`, `NOT`                                                   |

```sql
SELECT Sname, Sno, Ssex
FROM Student
WHERE Sname LIKE '刘%';
```

```sql
SELECT Cname, Cno
FROM Course
WHERE Cno LIKE '81__6';
/* 注意课程关系中课程号为固定长度，占5个字符大小 */
```

```sql
SELECT Sname, Sno, Ssex
FROM Student
WHERE Sname NOT LIKE '刘%';
```

```sql
SELECT *
FROM Product
WHERE description LIKE '%\_%' ESCAPE '\';
```

#### 3. ORDER BY子句的使用

```sql
SELECT Sno, Grade
FROM SC
WHERE Cno = '81003'
ORDER BY Grade DESC;
```

#### 4. 聚集函数

| 聚合函数                          | 功能说明                                     |
|-----------------------------------|----------------------------------------------|
| `COUNT(*)`                        | 统计元组个数（即行数）                       |
| `COUNT([DISTINCT \| ALL] <列名>)` | 统计一列中值的个数（默认 ALL，含重复；DISTINCT 去重） |
| `SUM([DISTINCT \| ALL] <列名>)`   | 计算一列值的总和（此列必须是数值型）         |
| `AVG([DISTINCT \| ALL] <列名>)`   | 计算一列值的平均值（此列必须是数值型）       |
| `MAX([DISTINCT \| ALL] <列名>)`   | 求一列值中的最大值                           |
| `MIN([DISTINCT \| ALL] <列名>)`   | 求一列值中的最小值                           |

当遇到空值时, 聚集函数中除了COUNT(*)以外, 其他聚集函数会忽略空值。

```sql
SELECT COUNT(DISTINCT Sno)
FROM SC;
```

#### 5. GROUP BY子句

GROUP BY 子句将查询结果按某一列或多列的值分组，值相等的为一组，如果是多列，则先对第一列的值分组，然后对每一组中的值按照第二列值分组，以此类推。结果集中如果有重复的列组值，则将其合并为一行输出。

```sql
SELECT Cno, COUNT(Sno)
FROM SC
GROUP BY Cno;
```

查询在 20192 学期选课超过 10 门的学生学号
```sql
SELECT Sno
FROM SC
WHERE Semester = '20192'
GROUP BY Sno
HAVING COUNT(*) > 10;
```

#### 6. LIMIT子句

```sql
SELECT Sno, AVG(Grade)
FROM SC
GROUP BY Sno
ORDER BY AVG(Grade) DESC
LIMIT 5 OFFSET 2;
/* 取 5 行数据，忽略前 2 行，之后为查询结果数据 */
```

### 3.3.2 连接查询

若一个查询同时涉及两个以上的表，则称之为连接查询.

#### 1. 等值与非等值连接查询

```sql
SELECT Student.*, SC.*
FROM Student, SC
WHERE Student.Sno = SC.Sno;
/* 将 Student 与 SC 中同一学生的元组连接起来 */
```

#### 2. 自然连接查询

把结果表目标列中重复的属性列去掉的等值连接查询则为自然连接查询

```sql
SELECT Student.Sno, Sname, Ssex, Sbirthdate, Smajor, Cno, Grade
FROM Student, SC
WHERE Student.Sno = SC.Sno;
```

#### 3. 符合条件连接查询

```sql
SELECT Student.Sno, Sname
FROM Student, SC
WHERE Student.Sno = SC.Sno 
  AND SC.Cno = '81002' 
  AND SC.Grade > 90;
/* 连接谓词 */        /* 其他选择条件 */
```

#### 4. 自身连接查询

连接操作不仅可以在两个表之间进行，也可以是一个表与其自身进行连接，此时的连接查询称为表的自身连接查询。

```sql
SELECT FIRST.Cno, SECOND.Cpno
FROM Course FIRST, Course SECOND
WHERE FIRST.Cpno = SECOND.Cno 
  AND SECOND.Cpno IS NOT NULL;
```

#### 5. 外连接查询

```sql
SELECT Student.Sno, Sname, Ssex, Sbirthdate, Smajor, Cno, Grade
FROM Student LEFT OUTER JOIN SC ON (Student.Sno = SC.Sno);
```

#### 6. 多表连接查询

```sql
SELECT Student.Sno, Sname, Cname, Grade
FROM Student, SC, Course
WHERE Student.Sno = SC.Sno 
  AND SC.Cno = Course.Cno;
```

### 3.3.3 嵌套查询

查询选修了课程号为 '81003' 的学生姓名
```sql
SELECT Sname                          /* 外层查询或父查询*/
FROM Student
WHERE Sno IN (
    SELECT Sno                        /* 内层查询或子查询*/
    FROM SC
    WHERE Cno = '81003'
);
```

#### 1. 带有IN谓词的子查询

## 3.4 数据更新

SQL的数据更新包括插入数据、删除数据和修改数据。

### 3.4.1 插入数据

#### 1. 插入新元组

```sql
INSERT INTO <表名> [ (<属性列1>[, <属性列2>]...) ]

```

其功能是将一个新元组插人指定表中。新元组的属性列1的值为常量1，属性列2的值为常量2, INTO 子句中没有出现的属性列，新元组在这些列上将取空值。

#### 2. 插入子查询结果

```sql
INSERT INTO <表名> [ (<属性列1>[, <属性列2>]...) ]
子查询;
```

举例如下

```sql
-- 第一步：创建新表 Smajor_age，用于存储专业名和平均年龄
CREATE TABLE Smajor_age (
    Smajor VARCHAR(20),
    Avg_age SMALLINT
);

-- 第二步：从 Student 表中按专业分组计算平均年龄，并插入到 Smajor_age 表中
INSERT INTO Smajor_age (Smajor, Avg_age)
SELECT 
    Smajor, 
    AVG(EXTRACT(YEAR FROM CURRENT_DATE) - EXTRACT(YEAR FROM Sbirthdate)) AS Avg_age
FROM Student
GROUP BY Smajor;
```

## 3.5 空值的处理

所谓空值就是“不知道”“不存在”或“无意义”的值. SQL语言中允许某些元组的某些属性在一定情况下取空值

#### 1. 空值的产生

```sql
INSERT INTO SC (Sno, Cno, Grade, Semester, Teachingclass)
VALUES ('20180006', '81004', NULL, '20211', NULL);

/* 
 * 说明：
 * - Grade = NULL       → 成绩暂为空（未考试或未录入）
 * - Teachingclass = NULL → 教学班暂未分配
 * 
 * 注意：NULL 表示“未知”或“不适用”，不是空字符串 '' 或 0！
 */
```

#### 2. 控制的判断

```sql
-- [例 3.83] 从 Student 表中找出漏填了数据的学生信息
SELECT *
FROM Student
WHERE Sname IS NULL 
   OR Ssex IS NULL 
   OR Sbirthdate IS NULL 
   OR Smajor IS NULL;
```

#### 3. 控制约束

- **NOT NULL 约束**  
  - 属性定义为 `NOT NULL`，则**不能取空值（NULL）**。  
  - **主码属性自动隐含 NOT NULL**：如 `SC(Sno, Cno)` 中 `Sno`、`Cno` 均不可为空；`Student(Sno)` 中 `Sno` 不可为空。

- **UNIQUE 约束与空值**  
  - 具有 `UNIQUE` 约束的属性**允许取 NULL**。  
  - **多个记录能否同时为 NULL？** SQL 标准未明确规定，取决于具体 DBMS：
    - 允许多个 NULL：Oracle、KingbaseES、PostgreSQL  
    - 不允许重复 NULL（即最多一个 NULL）：SQL Server  

#### 4. 空值的算术运算、比较运算和逻辑运算

```sql
SELECT Sno
FROM SC
WHERE Cno = '81001' 
  AND (Grade < 60 OR Grade IS NULL);
```

## 3.6 视图

- **定义**：视图是从一个或多个基本表（或其他视图）导出的**虚表**，数据库中仅存储其**定义（查询语句）**，不存储实际数据。
  
- **数据动态性**：视图中的数据始终来源于底层基表。**基表数据变化 → 视图查询结果自动同步更新**，如同“窗口”实时反映感兴趣的数据。

- **操作能力**：
  - 可像基表一样被 **SELECT 查询**
  - 可在视图基础上再定义新视图（嵌套视图）
  - 对视图的 **INSERT / UPDATE / DELETE 操作有限制**（取决于视图结构是否满足可更新条件，如是否含聚合、DISTINCT、JOIN 等）

- **核心价值**：
  - 提供多角度数据观察方式
  - 简化复杂查询
  - 实现逻辑数据独立性
  - 增强安全性（隐藏敏感字段/行）

### 3.6.1 定义视图

#### 1. 建立视图

SQL用''CREATE VIEW''语句来定义视图，基本格式如下：

```sql
CREATE VIEW <视图名> [ (<列名1>[, <列名2>]...) ]
AS <子查询>
[WITH CHECK OPTION];
```

例如:

```sql
CREATE VIEW High_Score (Sno, Cno, Grade) AS
SELECT Sno, Cno, Grade
FROM SC
WHERE Grade >= 90
WITH CHECK OPTION;
-- 后续若执行 UPDATE High_Score SET Grade=85 WHERE Sno='...' 会失败！
```

#### 2. 删除视图

```sql
DROP VIEW <视图名> [CASCADE];
```

- CASCADE：如果视图被其他视图或对象依赖，则同时删除这些依赖对象。(递归删除)

### 3.6.2 查询视图

```sql
-- [例 3.94] 在 IS_Student 视图中查找年龄 ≤ 20 的学生
SELECT Sno, Sbirthdate
FROM IS_Student
WHERE (EXTRACT(YEAR FROM CURRENT_DATE) - EXTRACT(YEAR FROM Sbirthdate)) <= 20;
```

### 3.6.3 更新视图

更新视图是指通过视图来插人、删除和修改数据。由于视图是不实际存储数据的虚表，因此对视图的更新最终要转换为对基本表的更新。像查询视图那样，对视图的更新操作也是通过视图消解，转换为对基本表的更新操作。

```sql
-- [例 3.97] 从用户角度：更新视图 IS_Student 中学号 '20180005' 的学生姓名
UPDATE IS_Student
SET Sname = '刘新奇'
WHERE Sno = '20180005';

-- 数据库系统自动将上述操作转换为对基表 Student 的更新：
UPDATE Student
SET Sname = '刘新奇'
WHERE Sno = '20180005' 
  AND Smajor = '信息管理与信息系统';  -- ← 关键！必须加上视图定义中的条件
```

### 3.6.4 视图的作用

#### 1. 视图能够对机密数据提供安全保护

#### 2. 视图对重构数据库提供了一定程度的逻辑独立性

#### 3. 视图能够简化用户的查询操作

#### 4. 视图使用户能以多种角度看待同一数据

# 第四章 数据库安全性

## 4.1 数据库安全性概述

数据库的安全性是指保护数据库, 以防不合法使用所造成的数据泄露、篡改或者破坏.

## 4.2 数据库安全性控制

在计算机系统中,安全措施是一级一级层层设置的.

1. **事前控制**  
   - 身份鉴别 + 入侵检测 → 阻止未授权访问，防患于未然。

2. **事中控制**  
   - SQL 层多级存取控制（自主/强制/推理控制）→ 用户仅能操作被授权数据。

3. **事后控制**  
   - 审计日志 + 异常行为分析 → 监控、追溯、拦截恶意操作，联动入侵检测系统。

4. **存储保护**  
   - 敏感数据与审计文档加密存储 → 防止静态数据泄露或篡改。

5. **传输保护**  
   - 客户端↔服务器间通信加密（如 SSL/TLS）→ 防止明文传输被窃听或中间人攻击。

### 4.2.1 用户身份鉴别

- 静态口令鉴别
- 动态口令鉴别
- 生物特征鉴别
- 智能卡鉴别
- 入侵检测

### 4.2.2 存取控制

#### 核心目标：
确保**只有授权用户**才能访问数据库，未授权者无法接近数据。

#### 两大组成部分：

1. **定义用户权限**
   - 管理员为用户分配对数据对象的操作权（如 SELECT、INSERT、UPDATE、DELETE）。
   - 权限信息存入**数据字典**，形成“安全规则”或“授权规则”。

2. **合法权限检查**
   - 用户每次发起操作请求时，DBMS 自动查数据字典 → 验证是否拥有对应权限。
   - 若无权限 → 拒绝执行，返回错误。

### 4.2.3 自主存取控制方法

大型数据库管理系统都支持自主存取控制方法, SQL标准也对自主存取提供支持, 这主要是通过'GRANT'和'REVOKE'语句来实现的.

用户权限是由两个要素组成的: 数据库对象

### 表 4.4：关系数据库中各类对象的存取权限汇总

| 对象类型         | 具体对象             | 可授予的操作权限（Privileges）                     |
|------------------|----------------------|----------------------------------------------------|
| **数据库和模式** | 数据库 / 模式        | `CREATE`                                           |
|                  | 基本表               | `CREATE TABLE`, `ALTER TABLE`                      |
|                  | 视图                 | `CREATE VIEW`                                      |
|                  | 索引                 | `CREATE INDEX`                                     |
| **数据**         | 基本表 和 视图       | `SELECT`, `INSERT`, `UPDATE`, `DELETE`, `REFERENCES`, `ALL PRIVILEGES` |
|                  | 属性列（字段级）     | `SELECT`, `INSERT`, `UPDATE`, `REFERENCES`, `ALL PRIVILEGES` |

### 4.2.4 授予与收回对数据的操作权限

#### 1. GRANT 语句

```sql
GRANT <权限> [, <权限>]...
ON <对象类型> <对象名> [, <对象类型> <对象名>]...
TO <用户> [, <用户>]...
[WITH GRANT OPTION];
```

- WITH GRANT OPTION: 允许被授权用户将自己拥有的权限继续授权给其他用户（即“二次授权”）。

#### 2. REVOKE 语句

```sql
REVOKE <权限> [, <权限>]...
ON <对象类型> <对象名> [, <对象类型> <对象名>]...
FROM <用户> [, <用户>]... [CASCADE | RESTRICT];
```
- CASCADE: 级联收回权限，即同时收回被授权用户再授权给其他用户的权限。
- RESTRICT: 限制收回权限，如果被授权用户已将权限授权给其他用户，则拒绝收回权限。

#### 3. 创建数据库的权限

CREATE USER 语句建立超级用户, 再由超级用户创建具有CREATE 数据库权限的用户

```sql
CREATE USER <username> [WITH]
[SUPERUSER | CREATEDB] | PASSWORD 'password';
```

### 4.2.5 数据库角色

数据库角色是被命名的一组与数据库操作相关的权限，角色是权限的集合。

#### 1. 创建角色

```sql
CREATE ROLE <角色名>
```

#### 2. 授予角色权限

```sql
GRANT <权限> [, <权限>]...
ON <对象类型> <对象名>
TO <角色> [, <角色>]...;
```

#### 3. 将角色授予用户

```sql
GRANT <角色1> [, <角色2>]...
TO <角色3> [, <用户1>]...
[WITH ADMIN OPTION];
```

- WITH ADMIN OPTION: 允许被授予角色的用户将该角色继续授予其他用户（即“二次授权”）。

#### 4. 收回角色权限

```sql
REVOKE <权限> [, <权限>]...
ON <对象类型> <对象名>
FROM <角色> [, <角色>]... [CASCADE | RESTRICT];
```

### 4.3 视图机制

-- 先建立视图 CS_Student

# 第五章 数据库完整性

数据库的完整性是指数据库数据的正确性和相容性, 数据的正确性是指数据库数据符合现实世界语意且反映当前的实际状况, 数据的相容性是指数据库同一对象在不同关系表中的数据是相同的, 一致的.

## 5.1 数据库完整性概述

为维护数据库的完整性，关系数据库管理系统必须能够实现以下功能。

- 1. 提供定义完整性约束的机制。

- 2. 提供检查完整性约束的方法.

- 3. 提供完整性的违约处理方法

## 5.2 实体完整性

### 5.2.1 定义实体完整性

关系模型的实体完整性在CREATE TABLE中用PRIMARY KEY定义.

创建“学生”表Student, 将Sno属性定义为主码.

```sql
-- 在列级定义主码
CREATE TABLE Student (
    Sno CHAR(8) PRIMARY KEY,      -- 学号为主键
    Sname CHAR(20) UNIQUE,        -- 姓名唯一
    Ssex CHAR(6),                 -- 性别
    Sbirthdate DATE,              -- 出生日期
    Smajor VARCHAR(40)            -- 专业
);
```

```sql
-- 在表级定义主码
CREATE TABLE Student (
    Sno CHAR(8),                  -- 学号
    Sname CHAR(20) UNIQUE,        -- 姓名唯一
    Ssex CHAR(6),                 -- 性别
    Sbirthdate DATE,              -- 出生日期
    Smajor VARCHAR(40),           -- 专业
    PRIMARY KEY (Sno)             -- 表级指定主键
);
```

### 5.2.2 实体完整性检查和违约处理

用PRIMARY KEY短语定义了关系的主码后, 每当用户对基本表插入一条记录或者对主码列进行更新操作时,会自动检查:

- 检查主码值是否唯一, 如果不唯一则拒绝插入或者修改.

- 检查主码的各个属性是否为空, 只要有一个为空就拒绝插入或者修改

## 5.3 参照完整性

### 5.3.1 定义参照完整性

关系模型的参照完整性在CREATE TABLE中用FOREIGN KEY短语定义哪些列尾外码, 用REFERENCES短语制定这些外码参照哪些表的主码.

```sql
-- [例 5.3] 定义 SC 中的参照完整性
CREATE TABLE SC (
    Sno CHAR(8),
    Cno CHAR(5),
    Grade SMALLINT,
    Semester CHAR(5),
    Teachingclass CHAR(8),

    PRIMARY KEY (Sno, Cno),                          -- 在表级定义实体完整性（复合主键）

    FOREIGN KEY (Sno) REFERENCES Student(Sno),       -- 在表级定义参照完整性：Sno 是外码，被参照表是 Student
    FOREIGN KEY (Cno) REFERENCES Course(Cno)         -- 在表级定义参照完整性：Cno 是外码，被参照表是 Course
);
```

#### 表 5.1 可能破坏参照完整性的情况及违约处理

| 被参照表 (如 Student)          | 参照片 (如 SC)             | 违约处理                     |
|-------------------------------|---------------------------|------------------------------|
| 可能破坏参照完整性 ←           | 插入元组                  | 拒绝                         |
| 可能破坏参照完整性 ←           | 修改外码值                | 拒绝                         |
| 删除元组 →                    | 可能破坏参照完整性         | 拒绝 / 级联删除 / 设置为空值   |
| 修改主码值 →                  | 可能破坏参照完整性         | 拒绝 / 级联修改 / 设置为空值   |

## 5.4 用户定义的完整性

### 5.4.1 属性上的约束

- 列值非空约束 (NOT NULL)

- 列值唯一约束 (UNIQUE)

- 列值检查约束 (CHECK)

```sql
CREATE TABLE SC (
    Sno CHAR(8),
    Cno CHAR(5),
    Grade SMALLINT CHECK (Grade >= 0 AND Grade <= 100),   -- Grade 取值范围 0~100
    Semester CHAR(5),
    Teachingclass CHAR(8),

    PRIMARY KEY (Sno, Cno),                                -- 复合主键：实体完整性

    FOREIGN KEY (Sno) REFERENCES Student(Sno),             -- 参照完整性：引用学生表
    FOREIGN KEY (Cno) REFERENCES Course(Cno)               -- 参照完整性：引用课程表
);
```

### 5.4.2 元组上的约束

同属性值限制相比, 元组级的限制可以设置不同属性之间取值的相互约束.

当学生的性别是男时, 其姓名不能是Ms.开头;
```sql
-- [例 5.9] 当学生的性别是“男”时，其姓名不能以 Ms. 打头。
CREATE TABLE Student (
    Sno CHAR(8),
    Sname CHAR(20) NOT NULL,
    Ssex CHAR(6),
    Sbirthdate DATE,
    Smajor VARCHAR(40),

    PRIMARY KEY (Sno),

    -- 定义了元组中 Sname 和 Ssex 两个属性值之间的约束条件
    CHECK (Ssex = '女' OR Sname NOT LIKE 'Ms. %')
);
```

## 5.5 完整性约束命名子句

SQL还在CREATE TABLE语句中提供了完整性约束命名子句CONSTRAINT, 用来对完整性约束命名, 从而可以灵活增加或者删除一个完整性约束

#### 1. 完整性约束命名子句格式

```sql
CONSTRAINT <完整性约束名><完整性约束>
```

<完整性约束>包括:
- NOT NULL
- UNIQUE
- CHECK
- PRIMARY KEY
- FOREIGN KEY

```sql
CREATE TABLE Student (
    Sno CHAR(8)
        CONSTRAINT C1 CHECK (Sno BETWEEN '1000000' AND '2999999'),   -- 学号范围约束

    Sname CHAR(20)
        CONSTRAINT C2 NOT NULL,                                     -- 姓名非空约束

    Sbirthdate DATE
        CONSTRAINT C3 CHECK (Sbirthdate > '1980-1-1'),              -- 出生日期晚于1980年

    Ssex CHAR(6)
        CONSTRAINT C4 CHECK (Ssex IN ('男', '女')),                 -- 性别只能为“男”或“女”

    Smajor VARCHAR(40),

    CONSTRAINT StudentKey PRIMARY KEY (Sno)                         -- 主键约束，命名为 StudentKey
);
```

#### 2. 删除完整性约束

```sql
ALTER TABLE Student
DROP CONSTRAINT C1;  -- 删除名为 C1 的完整性约束

```

## 5.6 断言

断言是对数据库中一个或多个关系表的元组之间的约束, 断言是数据库级的完整性约束, 其定义和检查都由数据库系统来完成.

#### 1. 创建断言

CREATE ASSERTION ASSE_SC_DB_NUM
CHECK (

        SELECT COUNT(*)
        FROM Course, SC
        WHERE SC.Cno = Course.Cno 
          AND Course.Cname = '数据库系统概论'
        GROUP BY Semester, Teachingclass
    )
);
```

```

## 5.7 触发器

### 5.7.1 定义触发器

SQL使用CREATE TRIGGER命令建立触发器, 其一般格式如下:

```sql
CREATE TRIGGER <触发器名>
    { BEFORE | AFTER } <触发事件> ON <表名>          -- 指明触发器激活的时间是在执行触发事件前或后
    REFERENCING NEW|OLD AS <变量>                   -- REFERENCING 指出引用的变量
    FOR EACH { ROW | STATEMENT }                    -- 定义触发器的类型，指明动作体执行的频率
    [ WHEN <触发条件> ] <触发动作体>;               -- 仅当触发条件为真时才执行触发动作体
```

- 触发事件：INSERT、DELETE、UPDATE
- 触发器类型: 触发器按照所触发动作的间隔尺寸可以分为行级触发器和语句级触发器
  - 行级触发器：FOR EACH ROW，指明触发器的动作体对每一行数据的修改都要执行一次。
  - 语句级触发器：FOR EACH STATEMENT，指明触发器的动作体在每次执行触发事件时只执行一次。
- 触发条件：WHEN子句中指定的条件表达式, 只有当触发事件发生时满足该条件表达式时, 触发器的动作体才会被执行.
- 触发动作体：触发器的动作体是一个SQL语句块, 该SQL语句块可以包含一个或多个SQL语句, 这些SQL语句在触发器被激活时执行.

```sql
-- [例 5.18] 当对表 SC 的 Grade 属性进行修改时，若分数增加了 10%，则记录到 SC_U 表中
CREATE TRIGGER SC_T
AFTER UPDATE ON SC                          -- 在 SC 表更新后触发
REFERENCING
    OLD AS OldTuple                         -- 引用更新前的整行数据
    NEW AS NewTuple                         -- 引用更新后的整行数据
FOR EACH ROW                                -- 行级触发器：每更新一行就执行一次
WHEN (NewTuple.Grade >= 1.1 * OldTuple.Grade)  -- 条件：新成绩 ≥ 旧成绩 × 1.1（即增加 ≥10%）
BEGIN
    INSERT INTO SC_U (Sno, Cno, OldGrade, NewGrade)
    VALUES (OldTuple.Sno, OldTuple.Cno, OldTuple.Grade, NewTuple.Grade);
END;
```

### 5.7.2 执行触发器

同一个表上的多个触发器激活时遵循以下执行顺序：
- 1. BEFORE 触发器先于 AFTER 触发器执行。
- 2. 同一时间点的多个触发器（如多个 BEFORE 触发器）按照创建顺序依次执行。
- 3. 行级触发器在语句级触发器之前执行。

### 5.7.3 删除触发器

```sql
DROP TRIGGER <触发器名> ON <表名>;
```

# 第六章 关系数据理论

## 6.1 问题的提出

#### 1. 要解决的核心问题

- 针对具体业务, 应该构造几个关系模式
- 每个关系模型应该包含哪些属性
- 如何判断一个数据库模式是“好”的?

#### 2. 关系模式的简化表示

完整形式化定义: $R(U, D, DOM,F)$

其中:
- $R$：关系模式的名称
- $U$：属性集，即关系模式中的属性集合
- $D$：域集，即关系模式中每个属性的取值范围
- $DOM$：域函数，定义了每个属性对应的域
- $F$：函数依赖集，定义了关系模式中属性之间的函数依赖关系

本章简化为 $R(U, F)$

第一范式(1NF): 关系的每个分量必须是不可分的数据项, 是关系模式的基本要求

#### 3. 数据依赖的核心概念

数据以来是关系属性之间的约束关系, 是现实世界属性间联系的抽象, 也是数据内在的语义体现

    - 最重要的两种类型: 函数依赖(FD) 和 多值依赖(MVD)

## 6.2 规范化

### 6.2.1 函数依赖

**定义** 给定关系模式 $R(U,F)$，$X$，$Y$ 是 $U$ 的子集。若对于 $R(U,F)$ 的任意一个可能的关系 $r$，$r$ 中不可能存在两个元组在 $X$ 上的属性值相等，而在 $Y$ 上的属性值不等，则称 $X$ 函数确定 $Y$ 或 $Y$ 函数依赖于 $X$，记作 $X \to Y$。

下面介绍一些术语和记号。

1.  $X \to Y$，但 $Y \not\subseteq X$，则称 $X \to Y$ 是非平凡的函数依赖（non-trival functional dependency）。
2.  $X \to Y$，但 $Y \subseteq X$，则称 $X \to Y$ 是平凡的函数依赖（trival functional dependency）。对于任一关系模式，平凡的函数依赖都是必然成立的，它不反映新的语义。若不特别声明，总是讨论非平凡的函数依赖。
3.  若 $X \to Y$，则 $X$ 称为这个函数依赖的决定属性组，也称为决定因素（determinant）。
4.  若 $X \to Y$，$Y \to X$，则记作 $X \leftrightarrow Y$。
5.  若 $Y$ 不函数依赖于 $X$，则记作 $X \nrightarrow Y$。

**定义** 在 $R(U,F)$ 中，如果 $X \to Y$，并且对于 $X$ 的任何一个真子集 $X'$，都有 $X' \nrightarrow Y$，则称 $Y$ 对 $X$ **完全函数依赖**（full functional dependency），记作 $X \stackrel{F}{\rightarrow} Y$。

**定义** 在 $R(U,F)$ 中，如果 $X \to Y$（$Y \not\subseteq X$），$Y \nrightarrow X$，$Y \to Z$，$Z \not\subseteq Y$，则称 $Z$ 对 $X$ **传递函数依赖**（transitive functional dependency），记为 $X \stackrel{T}{\rightarrow} Z$。

### 6.2.2 码

**定义** 设 $K$ 为 $R(U,F)$ 中的属性或属性组，若 $K \stackrel{F}{\rightarrow} U$，则 $K$ 为 $R$ 的候选码。

如果 $U$ 部分函数依赖于 $K$，即 $K \stackrel{P}{\rightarrow} U$，则 $K$ 称为超码。

### 6.2.3 范式

对于各种范式之间的关系有
$$4NF \subset BCNF \subset 3NF \subset 2NF \subset 1NF$$成立

一个低一级范式的关系模式通过模式分解（schema decomposition）可以转换为若干个更高级别范式的关系模式的集合，这种过程就叫规范化。

### 6.2.4 2NF

如果关系模式 $R$ 的每一个分量是一个不可分的数据项，$R$ 是满足最低要求的关系模式，则
$$R \in 1NF。$$

**定义 6.6** 若 $R \in 1NF$，且每一个非主属性完全函数依赖于任何一个候选码，则 $R \in 2NF$。

### 6.2.5 3NF

**定义** 设关系模式 $R(U,F) \in 1NF$，若 $R$ 中不存在这样的码 $X$，属性组 $Y$ 及非主属性 $Z$（$Y \nsupseteq Z$），使得 $X \to Y$，$Y \to Z$ 成立，$Y \nrightarrow X$，则称 $R(U,F) \in 3NF$。

由定义 6.7 可以证明，若 $R \in 3NF$，则每一个非主属性既不传递依赖于码，也不部分依赖于码。也就是说，可以证明如果 $R$ 属于 3NF，则必有 $R$ 属于 2NF。

### 6.2.6 BCNF

BCNF(Boyce Codd Normal Form)是由 Boyce 与 Codd 提出的，比上述 3NF 又进了一步，通常认为 BCNF 是修正的第三范式，有时也称为扩充的第三范式。

**定义** 关系模式 $R(U,F) \in 1NF$，若 $X \to Y$ 且 $Y \not\subseteq X$ 时 $X$ 必含有码，则 $R(U,F) \in \text{BCNF}$。
也就是说，关系模式 $R(U,F)$ 中，若每一个决定因素都包含码，则 $R(U,F) \in \text{BCNF}$。

由 BCNF 的定义可以得到结论，一个满足 BCNF 的关系模式有：
① 所有非主属性对每一个码都是完全函数依赖。
② 所有主属性对每一个不包含它的码也是完全函数依赖。
③ 没有任何属性完全函数依赖于非码的任何一组属性。

由于 $R \in \text{BCNF}$，按定义排除了任何属性对码的传递依赖与部分依赖，所以 $R \in 3NF$。严格的证明留给读者完成。但是若 $R \in 3NF$，$R$ 未必属于 BCNF。

### 6.2.7 多值依赖

**定义** 给定关系模式 $R(U,F)$，$X$、$Y$、$Z$ 是 $U$ 的子集，并且 $Z = U - X - Y$。关系模式中多值依赖 $X \twoheadrightarrow Y$ 成立，当且仅当对 $R(U,F)$ 的任一关系 $r$ 给定的一对 $(x,z)$ 值，有一组 $Y$ 的值，这组值仅仅决定于 $x$ 值而与 $z$ 值无关。

若 $X \twoheadrightarrow Y$，而 $Z = \varnothing$，即 $Z$ 为空，则称 $X \twoheadrightarrow Y$ 为平凡的多值依赖。即对于 $R(X,Y)$，如果有 $X \twoheadrightarrow Y$ 成立，则 $X \twoheadrightarrow Y$ 为平凡的多值依赖。

多值依赖具有以下性质：

- 多值依赖具有对称性。即若 $X \twoheadrightarrow Y$，则 $X \twoheadrightarrow Z$，其中 $Z=U-X-Y$。
从例 6.10 容易看出，因为每个保管员保管所有商品，同时每种商品被所有保管员保管，显然若 $W \twoheadrightarrow S$，必然有 $W \twoheadrightarrow C$。

- 多值依赖具有传递性。即如果 $XYZ=U$，$X \twoheadrightarrow Y$，$Y \twoheadrightarrow Z$，则 $X \twoheadrightarrow Z-Y$。

- 函数依赖可以看作是多值依赖的特殊情况，即若 $X \to Y$，则 $X \twoheadrightarrow Y$。这是因为当 $X \to Y$ 时，对 $X$ 的每一个值 $x$，$Y$ 有一个确定的值 $y$ 与之对应，故 $X \twoheadrightarrow Y$。

- 若 $X \twoheadrightarrow Y$，$X \twoheadrightarrow Z$，则 $X \twoheadrightarrow YZ$。

- 若 $X \twoheadrightarrow Y$，$X \twoheadrightarrow Z$，则 $X \twoheadrightarrow Y \cap Z$。

- 若 $X \twoheadrightarrow Y$，$X \twoheadrightarrow Z$，则 $X \twoheadrightarrow Y-Z$，$X \twoheadrightarrow Z-Y$。

### 6.2.8 4NF

**定义** 关系模式 $R(U,F) \in 1NF$，如果对于 $R$ 的每个非平凡多值依赖 $X \twoheadrightarrow Y$（$Y \not\subseteq X$），$X$ 都含有码，则称 $R(U,F) \in 4NF$。

## 6.3 数据依赖的公理系统

**定义** 给定关系模式 $R(U,F)$，其任何一个关系 $r$，若函数依赖 $X \to Y$ 都成立（即 $r$ 中任意两元组 $t$、$s$，若 $t[X] = s[X]$，则 $t[Y] = s[Y]$），则称 $F$ 逻辑蕴涵 $X \to Y$。

为了从一组函数依赖求得蕴涵的函数依赖，例如已知函数依赖集 $F$，要问 $X \to Y$ 是否为 $F$ 所蕴涵，就需要一套推理规则，这组推理规则是 1974 年首先由 Armstrong 提出来的。

**Armstrong 公理系统（Armstrong's axiom）** 设 $U$ 为属性组全集，$F$ 是 $U$ 上的一组函数依赖，于是有关系模式 $R(U,F)$，对 $R(U,F)$ 来说有以下的推理规则：

- **A1 自反律（reflexivity rule）**：若 $Y \subseteq X \subseteq U$，则 $X \to Y$ 为 $F$ 所蕴涵。
- **A2 增广律（augmentation rule）**：若 $X \to Y$ 为 $F$ 所蕴涵，且 $Z \subseteq U$，则 $XZ \to YZ$ 为 $F$ 所蕴涵。
- **A3 传递律（transitivity rule）**：若 $X \to Y$ 及 $Y \to Z$ 为 $F$ 所蕴涵，则 $X \to Z$ 为 $F$ 所蕴涵。

根据 A1、A2、A3 这三条推理规则，可以得到下面三条很有用的推理规则。

- **合并规则（union rule）**：由 $X \to Y$，$X \to Z$，有 $X \to YZ$。
- **伪传递规则（pseudo transitivity rule）**：由 $X \to Y$，$WY \to Z$，有 $XW \to Z$。
- **分解规则（decomposition rule）**：由 $X \to Y$ 及 $Z \subseteq Y$，有 $X \to Z$。

根据合并规则和分解规则，很容易得到这样一个重要事实。

**引理** $X \to A_1A_2\cdots A_k$ 成立的充分必要条件是 $X \to A_i$ 成立（$i=1,2,\cdots,k$）。

**定义** 在关系模式 $R(U,F)$ 中为 $F$ 所逻辑蕴涵的函数依赖的全体叫作 $F$ 的闭包（closure），记为 $F^+$。

人们把自反律、传递律和增广律称为 Armstrong 公理系统。Armstrong 公理系统是有效的、完备的。其中，有效性是指由 $F$ 出发，根据 Armstrong 公理系统推导出来的每一个函数依赖一定在 $F^+$ 中；完备性是指 $F^+$ 中的每一个函数依赖，必定可以由 $F$ 出发根据 Armstrong 公理系统推导出来。

## 6.4 保持函数依赖的模式分解

**定义** 关系模式 $R(U,F)$ 的一个分解是指

$$\rho = \{ R_1(U_1,F_1), R_2(U_2,F_2), \dots, R_n(U_n,F_n) \}$$

其中 $U = \bigcup_{i=1}^{n} U_i$，并且没有 $U_i \subseteq U_j$，$1 \leqslant i \neq j \leqslant n$，$F_i$ 是 $F$ 在 $U_i$ 上的投影，即

$$F_i = \{ X \to Y \mid X \to Y \in F^+ \land XY \subseteq U_i \}$$

**定义** $\rho = \{ R_1(U_1,F_1), R_2(U_2,F_2), \dots, R_k(U_k,F_k) \}$ 是关系模式 $R(U,F)$ 的一个保持函数依赖的分解，如果

$$F^+ = \left( \bigcup_{i=1}^{k} F_i \right)^+$$

**算法 6.2（合成法） 转换为 3NF 的保持函数依赖的分解。**

① 对 $R(U,F)$ 中的函数依赖集 $F$ 进行极小化处理（处理后得到的依赖集仍记为 $F$）。

② 找出所有不在 $F$ 中出现的属性（记为 $U_0$），构成一个关系模式 $R_0(U_0,F_0)$。把这些属性从 $U$ 中去掉，剩余的属性仍记为 $U$。

③ 若有 $X \to A \in F$，且 $XA = U$，则 $\rho = \{ R \}$，算法终止。

④ 否则，对 $F$ 按具有相同左部的原则分组（假定分为 $k$ 组），每一组函数依赖所涉及的全部属性形成一个属性集 $U_i$。若 $U_i \subseteq U_j (i \neq j)$ 就去掉 $U_i$。由于经过了步骤②，故 $U = \bigcup_{i=1}^{k} U_i$，于是 $\rho = \{ R_1(U_1,F_1), \dots, R_k(U_k,F_k) \} \cup R_0(U_0,F_0)$ 构成 $R(U,F)$ 的一个保持函数依赖的分解，并且每个 $R_i(U_i,F_i)$ 均属于 3NF。这里 $F_i$ 是 $F$ 在 $U_i$ 上的投影。

#### 3NF合成法 通俗步骤概括

1.  **先理干净规矩**：把所有"谁决定谁"的关系拆成最细的一对一，删掉重复、多余的规矩
2.  **拎出孤家寡人**：找出那些既不决定别人、也不被别人决定的属性，单独成表
3.  **检查特殊情况**：如果整个表就剩"一个老板管一个员工"，直接结束不用拆
4.  **按老板分组建表**：把同一个老板管的所有员工打包成一个部门（小表），删掉被其他部门完全包含的小部门
